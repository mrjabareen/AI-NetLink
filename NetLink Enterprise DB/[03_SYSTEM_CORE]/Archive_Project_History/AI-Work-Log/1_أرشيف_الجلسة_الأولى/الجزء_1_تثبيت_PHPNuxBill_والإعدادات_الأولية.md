# الجزء الأول: تثبيت PHPNuxBill والإعدادات الأولية
## توثيق مرحلة التأسيس للمشروع والمحادثات الأولى

هذا الجزء يوثق بداية رحلتنا في تجهيز نظام الفوترة NetLink والوصول به إلى مرحلة الاستقرار البرمجي.

---

### **[23-03-2026] - بداية المحادثة: تثبيت النظام**

**المدير (رسالة صوتية مُفرغة بدقة):** 
> "أريد البدء في تثبيت نظام PHPNuxBill على حاوية LXC في بروكس موكس (Proxmox). المواصفات المطلوبة هي Debian 12 مع تثبيت كافة الأدوات الأساسية والـ Web Server والـ PHP اللازمين لتشغيل البرنامج."

**Antigravity (الرد التقني):**
لقد بدأنا بإنشاء الحاوية بالمواصفات التالية:
- **نظام التشغيل:** Debian 12 Bookworm.
- **التجهيزات:** Apache2, MariaDB-Server, PHP 8.2 مع كافة الامتدادات الحيوية (curl, gd, mbstring, xml, mysql).
- **التنزيل:** استخدام Git لسحب الكود المصدري لـ PHPNuxBill ووضعه في `/var/www/html`.

---

### **[23-03-2026] - معالجة ثغرة الـ Cron Job**

**المدير (رسالة نصية):**
> "توجد رسالة خطأ حمراء بخصوص الكرون (Cron not configured) مع أنني قمت بتثبيته!"

**Antigravity (التحليل والحل):**
اكتشفنا أن البرنامج يحاول تشغيل الكرون من مسار خاطئ. قمنا بتعديل المسار في الـ `crontab` وجعلناه كالتالي ليعمل بنجاح:
```bash
* * * * * cd /var/www/html/system && /usr/bin/php cron.php > /dev/null 2>&1
```
*(ملاحظة: كان لابد من تنفيذ الأمر من داخل مجلد system ليتمكن من قراءة ملف init.php).*

---

### **[23-03-2026] - حل مشكلة الـ HTTPS والـ NPM**

**المدير (رسالة صوتية):**
> "الموقع لا يفتح بشكل صحيح عبر التشفير (HTTPS) وتصميم الموقع (CSS) يظهر بشكل محطم عند الدخول عليه من خلال الدومين الخاص بي."

**Antigravity (الحل البرمجي):**
بسبب وجود السيرفر خلف (Nginx Proxy Manager)، لم يتمكن نظام NuxBill من التعرف تلقائياً على البروتوكول الصحيح. قمنا بفتح ملف `config.php` وتغيير الإعداد اليدوي ليكون ثابتاً:
```php
$_app_url = 'https://net.netlinkps.com';
```
بعد هذا التعديل، تم حل مشكلة التصميم بالكامل.

### **[27-03-2026] - خطة المزامنة وحل مشاكل الداتا بيس**

**المدير (رسالة صوتية):**
> "أواجه مشكلة (No valid table found) في إضافة استهلاك البيانات، والبرنامج لا يتعرف على جدول radacct رغم وجوده في الداتا بيس."

**Antigravity (الحل التقني):**
قمنا بإجراء "عملية جراحية" وتعديل ملفات الإضافة (`data_usage_admin.php` و `data_usage_user.php`) لإجبار النظام على تجاهل دالة التحقق الفاشلة `isTableExist` والبدء فوراً في قراءة جدول `radacct` الحقيقي.

**الأوامر المنفذة للتثبيت السريع:**
```bash
sed -i "s/if (isTableExist(\$table)) {/if (true) { \$table = 'radacct'; /g" /var/www/html/system/plugin/data_usage_admin.php
```

بهذا نكون قد أغلقنا المرحلة الأولى بنجاح واستقرار.
