# خطة اختبار وإدارة أجهزة Cudy (CPE Management Plan)

## مقدمة
تهدف هذه الخطة إلى توفير دليل شامل وعملي لاختبار وإدارة موديلات Cudy المتوفرة (AX3000, AX1500, WR1300, WR1200, RE1200). 
بناءً على التوجيهات، يتم التركيز في هذه الخطة على كيفية تطبيق الإعدادات المطلوبة سواء عبر واجهة الويب (Web UI) أو **برمجياً** باستخدام بروتوكولات الإدارة المتاحة لمزودي الخدمة مثل TR-069 (CWMP) وواجهات سطر الأوامر (SSH/UCI - كون أجهزة Cudy مبنية في جوهرها على بيئة OpenWrt) واستدعاءات HTTP API.

---

## 1. إدارة الوصول والتحكم عن بُعد (Remote Access)

### أ. تفعيل الـ Remote Management:
- **التطبيق عبر واجهة المتصفح (Web UI):**
  - الانتقال إلى `Advanced Settings` > `Remote Management`.
  - تفعيل `Remote Web Management`.
  - تحديد المنفذ (Port) إلى `8080`.
  - السماح للوصول من كافة عناوين IP (`0.0.0.0`) أو تحديده بنطاق IP الإدارة الخاص بالـ ISP فقط لتوفير أمان إضافي.
- **التطبيق البرمجي (عبر TR-069 من نظام ACS كـ GenieACS):**
  - يمكن ضبط البارامترات برمجياً لجميع الأجهزة دون تدخل بشري عن طريق تعديل القيم التالية:
    - تفعيل الوصول: `InternetGatewayDevice.UserInterface.RemoteAccess.Enable` بالقيمة `true` (أو `1`).
    - تعيين المنفذ: `InternetGatewayDevice.UserInterface.RemoteAccess.Port` بالقيمة `8080`.
- **التطبيق البرمجي (عبر HTTP API / سكريبت خارجي):**
  - يمكن كتابة سكريبت (Bash أو Python) يقوم بتسجيل الدخول للحصول على `Token` (من مسار `/cgi-bin/luci/`)، ثم إرسال طلب `HTTP POST JSON` به إعدادات تفعيل الـ Remote Management وحفظها.

### ب. إعدادات الـ Cloud و Cudy App:
- **التطبيق التقليدي:**
  - ربط الراوتر بحساب Cudy للتحكم من خارج شبكة واسم الـ (SSID) عبر التطبيق المخصص للهاتف.
- **المنظور البرمجي لمزود الخدمة (ISP Point of View):**
  - في شبكات مزودي الخدمة الاحترافية، يُفضل دائماً الاعتماد على خادم التحكم الآلي (ACS Server) الخاص بالمزود بدلاً من الاعتماد على سيرفرات سحابة Cudy العامة. عبر الـ ACS، يمكنك الحصول على تحكم كامل (إعادة التشغيل Reboot، تغيير اسم شبكة واي فاي SSID، وكلمات المرور) باستخدام نداءات من نظام الـ Billing (مثل PHPNuxBill أو Mikrotik API) لتتواصل برمجياً مع سيرفر ה ACS.
  - مسار الـ SSID الخاص بالـ TR-069: `InternetGatewayDevice.LANDevice.1.WLANConfiguration.1.SSID`

## 2. توجيه المنافذ والخدمات (Port Forwarding & DMZ)

### أ. فتح البورتات (Port Forwarding):
- **السيناريو:** السماح بالوصول من خارج الشبكة إلى جهاز DVR داخلي.
- **تطبيق Web UI:** `Advanced` > `Port Forwarding` وإضافة القاعدة لـ IP الـ DVR والمنفذ المخصص.
- **التطبيق البرمجي (عبر TR-069):**
  - إنشاء `Object` جديد للـ Port Mapping في المسار: 
    `InternetGatewayDevice.WANDevice.1.WANConnectionDevice.1.WANIPConnection.1.PortMapping.`
  - تعيين القيم برمجياً:
    - `PortMappingEnabled` = `true`
    - `ExternalPort` = (منفذ الشبكة الخارجية مثل 8000)
    - `InternalPort` = (منفذ الـ DVR المحلي مثل 8000)
    - `InternalClient` = (الـ IP الداخلي للـ DVR مثل `192.168.1.100`)
- **التطبيق البرمجي (عبر SSH / OpenWrt UCI):**
  ```bash
  uci add firewall redirect
  uci set firewall.@redirect[-1].name='DVR_Forward'
  uci set firewall.@redirect[-1].src='wan'
  uci set firewall.@redirect[-1].src_dport='8000'
  uci set firewall.@redirect[-1].dest='lan'
  uci set firewall.@redirect[-1].dest_ip='192.168.1.100'
  uci set firewall.@redirect[-1].dest_port='8000'
  uci commit firewall
  /etc/init.d/firewall reload
  ```

### ب. تفعيل الـ DMZ:
- **التجربة والاختبار:** تحديد جهاز مستضيف Host (مثلاً `192.168.1.50`) وإضافته لمنطقة الـ DMZ لاختبار كفاءة جدار الحماية وعزل أو كشف الأجهزة.
- **التطبيق البرمجي:** عبر الـ ACS بتفعيل مدخل الـ DMZ وتحديد الـ `DMZHostIPAddress`.

## 3. نظام المش الذكي (Cudy Mesh Setup)

### أ. الربط اللاسلكي:
- **ربط الأجهزة:** 
  - الخطوة الفنية هي الضغط على زر WPS / Mesh Button على جهاز (الماستر) لمدة ثلاث ثوانٍ، وتكرار نفس الخطوة على الموديلات الداعمة مثل RE1200 خلال دقيقتين ليتم المزامنة تلقائياً.
- **التطبيق البرمجي:**
  - في حال وجود وصول SSH للراوتر الرئيسي، يمكن تفعيل البث للاقتران برمجياً بإرسال أمر (Trigger) للـ WPS (مثال أجهزة OpenWrt تستخدم `ubus call wps start`).

### ب. اختبار الـ Roaming:
- التأكد من الانتقال السلس دون انقطاع، يتم من خلال الاعتماد على تقنيات `802.11k/v/r`. 
- **طريقة الاختبار الفنية:**
  - ربط لابتوب محمول وعمل عملية مستمرة: `ping 8.8.8.8 -t`
  - التنقل السريع بين النقاط (Nodes) ومراقبة معدل الضياع (Packet loss). لا يجب أن يتجاوز الانقطاع بضعة مئات من الميلي ثانية (أقل من 50ms إذا كان Roaming يعمل بكفاءة).
  - يمكن مراقبة اتصال الجهاز برمجياً في الـ Master Node بقراءة سجلات الاتصال (Syslog) لرؤية عملية الـ (Re-association).

## 4. إدارة المشتركين والسرعات

### أ. التحكم بالأجهزة المتصلة (Blacklist):
- **سيناريو החظر:** منع الوصول لجهاز بهاتف المستخدم من الشبكة اللاسلكية (Wi-Fi MAC Filtering).
- **التطبيق البرمجي (MAC Filter عبر TR-069):**
  - تحديد حالة الفلترة لـ `MACAddressControlEnabled`.
  - إضافة الـ MAC Address غير المرغوب فيه للقائمة برمجياً وتحديد نوع القاعدة إلى (Deny أو Blacklisted).
- **التطبيق البرمجي (عبر SSH/UCI):**
  ```bash
  uci set wireless.@wifi-iface[0].macfilter='deny'
  uci add_list wireless.@wifi-iface[0].maclist='00:1E:8C:11:22:33'
  uci commit wireless
  wifi reload
  ```

### ب. تحديد السرعات و الـ Bandwidth Control:
- **تحكم من الراوتر:** التوجه لقائمة `Bandwidth Control` في لوحة الإدارة وتحديد الحد الأعلى لسرعتي الرفع والتحميل لجهاز معين (IP أو MAC).
- **نصيحة المنظور الشبكي لـ ISP:** بالرغم من توفر هذه الميزة على راوترات Cudy (CPE)، إلا أن الممارسة البرمجية الاحترافية هي التحكم في سرعات المستخدمين مركزياً عبر جهاز المايكروتيك (Mikrotik) الخاص بمزود الخدمة (باستخدام ردود Radius/PPPoE Profile Bandwidth) لتوفير دقة أعلى وتقليل استهلاك موارد المعالج لراوتر المشترك.

### ج. تحديث السوفتوير (Firmware):
- **التحديث عن بُعد (Mass Upgrade برمجياً):**
  - لضمان كفاءة أداء تقنيات Wi-Fi 6 و Mesh، يتم استخدام نظام الإدارة TR-069 لإرسال أمر `Download RPC` لعدد كبير من राواتر المشتركين في آنٍ واحد.
  - يتم تزويد الراوتر برابط (URL) للنسخة المحدثة (Firmware Binary) المتواجدة على سيرفر الدعم الفني للشركة (مثلاً: `http://10.0.0.10/firmware/cudy-ax3000-v2.bin`).
  - يمكن جدولة هذا الأمر من خلال سيرفر الـ ACS (مثلاً GenieACS) ليتم التنفيذ في الساعة 3 أو 4 فجراً لتقليل التأثير السلبي على عملة المتصلين.

---
**الخلاصة التوجيهية (التكامل البرمجي مع أنظمة الـ ISP):**
أجهزة Cudy مناسبة جداً لبيئات مزودي الخدمة بسبب دعمها لبروتوكول CWMP (TR-069). 
لتحقيق أقصى استفادة "برمجية"، يُفضل تجهيز خادم ACS متصل بلوحة التحكم لديكم. بهذه الطريقة، يمكن لأداة النظام من خلال الـ API (مثل سكربتات PHP أو Python مرتبطة بالبيلينج سيستم) أن ترسل أوامر للـ ACS، وبدوره يقوم الـ ACS تلقائياً وضمنياً بتعديل إعدادات الراوتر Cudy الخاص بالزبون فوراً دون الحاجة للدخول الفردي عبر متصفح الويب.
